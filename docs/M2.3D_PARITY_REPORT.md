# M2.3e XAU Parity Report

## Status

Issue #22 is no longer data-blocked. The live XAU parity failure was a
timeframe mismatch in the Python strategy port.

PR: `M2.3e: M5 aggregation for XAU parity`.

## Root Cause

The MT5 bridge evaluates XAU auto-trading on M5 bars:

- `FinRobotBridgeEA.mq5` line 21 sets `AutoTimeframe = PERIOD_M5`.
- Lines 751-777 read EMA, RSI, MACD, and ATR buffers from `AutoTimeframe`.
- Lines 813-814 compare the forming M5 close against the previous closed M5
  high/low for `ATR_impulse`.

The Python backtest strategies were evaluating those same indicators directly
on M1 bars. That shifted XAU decisions away from the live EA windows and the
live test failed before this fix:

- Before: `live XAU parity: 0/13 matched (0.00%)`.

## Fix

`XauM5RollingFeatureState` now aggregates incoming M1 bars into a forming M5
bar, commits closed M5 bars, and previews indicators against the forming M5
bar. XAU ATR impulse, quick momentum, and gated strategy paths use this M5
state.

The ATR impulse port also checks the M1 high/low for the first intraminute tick
that would have satisfied the EA's M5 `OnTimer()` condition, while the PDA/SMC
gate evaluates at that trigger price. The DuckDB parity helper now loads MT5
TSV timestamps as broker wall-clock epochs, matching the ack timestamps.

## Current Results

- Live XAU parity: `19/19 matched (100.00%)`.
- May real-data self-test: `9/9 matched (100.00%)`.
- Strategy unit coverage still checks breakout, cross, RSI ceiling/floor, and
  PDA/SMC gate behavior on M1 series generated from known M5 bars.
